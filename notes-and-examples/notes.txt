# Project vs application
In Django a website would be called a project. 
Applications are some parts of the website - e.g. blog, wiki, or forum.
The applications can be used by other projects
To create a Blog application: python manage.py startapp blog

# Data Models
Model is a Python class that subclasses django.db.models.Model where each attribute is a database field
A table is created for each model in models.py file
Djago provides various fields for data models: https://docs.djangoproject.com/en/6.0/ref/models/fields/
Custome model fields can also be created: https://docs.djangoproject.com/en/6.0/howto/custom-model-fields/

# Create a database table for the Model
!!Reminder!!: In Django a Website is a Project, a Blog is an App, a Blog Post is a data Model
1) Create initial migration for the Post data model, in the root dir: $ python manage.py makemigrations blog
^^ for migration to work, ensure that the app is activated, otherwise I will get an error: No installed app with label 'blog'.
^^ when the initial migration is done, a file is created in the blog directory, the file is called 0001_initial.py
^^ 0001_initial.py specifies dependencies and operations within the Migration class
2) Run this to inspect the output: $ python manage.py sqlmigrate blog 0001
3) Sync the db with the new data model: $ python manage.py migrate

# Creating administration site to manage blog posts
Django has a build-in admin interface, use out of box or can be configured to suit
INSTALLED_APPS already includes django.contrib.admin
1) Creating a superuser who will manage the admin site: 
1-1) $ python manage.py createsuperuser
1-2) Start development server: $ python manage.py runserver
1-3) Open: http://127.0.0.1:8000/admin
1-4) Log in and Be super happy that it all worked well :)
1-5)Group and User models in the admin page are located in django.contrib.auth
2) Add model(s) to the administration site:
2-1) Edit admin.py with the below code:
    from django.contrib import admin
    from .models import Post
    admin.site.register(Post)
2-2) Refresh http://127.0.0.1:8000/admin and the Blog part with the Post template would appear 
2-3) Open the Post template: the form will be there with form fields, all ready to fill in. 
-- Form datafields are from Post class located in mysite/blog/models.py
-- Form save buttons "Save and add another", "Save and continue editing", "SAVE" are out-of-the box buttons from Django admin interface - ModelAdmin.save_as and ModelAdmin.save_as_continue
https://docs.djangoproject.com/en/dev/ref/contrib/admin/#overriding-admin-templates
The buttons can be overriden, e.g.
https://stackoverflow.com/questions/13101281/in-django-admin-how-can-i-hide-save-and-continue-and-save-and-add-another-butto

# Database management
Django comes with db abstraction API which helps to create, retrieve, delete and update objects
ORM/Object-Relational-Mapper is compatible with MySQL, PostgreSQL, SQLite, Oracle and MariaDB
Django ORM is based on QuerySets that are requeries to get objects from the database
Database for the project can be identified in settings.py under DATABASES
DOCS:
https://docs.djangoproject.com/en/6.0/ref/models/
https://docs.djangoproject.com/en/6.0/topics/db/models/
"objects" is a default manager of each model. "object" gets all objects in the database. it can be customised in models.py
To retrieve objects: Post.objects.all()
To filter: Post.objects.filter(publish_year=2026, author='admin')
To exlude: Post.objects.filter(publish_year=2026).exclude(title_startswith='Who')
To delete: 
>>> post = Post.objects.get(id=8)
>>> post.delete()


# Building views of blog application
In Django a view is a Python function. It receives a web request and returns a web response.
'View' contains the logic of the response. It renders a template with variables passed to it. 
The view returns HTTP response together with the rendered output. The views are created in views.py inside blog post application
Steps:
-- Create application views
-- Define URL pattern for each view
    * URL patterns is made of a string patter, a view and as an option a name that can be used across the whole project
    * Django runs through each URL pattern, if requested URL matches then it stops at that URL
    * After that, "Django imports the view of the matching URL pattern, executes it, passes an instance of HttpRequest class and the keyword or positional argument" (p.28)
    * Canonical URLs are used as the main URL for the model (e.g. Blog post), get_absolute_url() are added to the model
-- Create HTML templates to render data that views generate
    * templates structure:
    templates/
        blog/
        base.html
            post/
                list.html
                detail.html
    * templates are written in HTML and Django templates language
    https://docs.djangoproject.com/en/6.0/ref/templates/language/
